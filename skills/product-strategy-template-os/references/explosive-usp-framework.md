# 爆发力 USP 框架

本文件用于第 6 章产品规划，防止 pipeline 只选出“最稳妥、最容易落地”的方向，却丢掉真正有爆品想象力的产品命题。

核心原则：

```text
公司能力不是静态边界。
好的 USP 可以要求公司长出相邻能力，
但不能要求公司跳进特别难、特别重、不可控的新系统。
```

## 1. 什么时候必须使用

在以下场景中，第 6 章必须先运行本框架，再进入产品架构、测试验证、规格参数、定价利润和工业设计：

- 已经有多个机会点，但主赛道显得平凡。
- 原评分模型偏向 `company_fit`、`easy_to_build`、`first_eye_visible`，但用户认为想象空间不够。
- 产品需要从“功能卖点”升级成“消费者一眼有感觉的场景命题”。
- 公司愿意为了更强 USP 做相邻能力成长。
- 竞品已经满足了常规利益点，必须挖更锋利的用户瞬间。

## 2. 输入

必须输入：

- 前面章节的竞品、用户、评论、营销和社媒结论。
- 已有机会点或赛道候选。
- 用户评论中的购买驱动和痛点主题。
- 竞品为什么有人买、为什么有人失望。
- 公司能力包和可成长边界。

推荐输入：

- Amazon 主图、视频、A+、五点描述、coupon、价格、评论数、评论质量。
- 评论样本，最好 250 条以上有效评论，并标注品牌/星级/场景。
- 社媒上同类人群的生活方式痛点、痒点、爽点。
- 初步 BOM / 关键元器件 / 工艺边界。

缺失数据必须标注：

```text
not_collected
manual_required
not_available_in_current_session
unknown
```

## 3. 公司能力边界

第 6 章不能把公司能力理解成“现在只能做什么”。默认分三层：

### 3.1 必须利用的核心能力

- 当前工艺和 know-how 已经覆盖。
- 可以形成真实产品价值，而不是页面包装。
- 例如灯具公司在光学、结构、CMF、品质制造、研发生产一体化上的能力。

### 3.2 可以为了 USP 拓展的相邻能力

- 超出现有能力一点点，但开发难度可控。
- 可以通过 ODM、外部模块、小批量样机、工程验证补齐。
- 成立后再考虑内部投人、投设备、沉淀 know-how。
- 例如空间光效曲线、本机 routine 引擎、声音模块、实体交互、可靠性长测。

### 3.3 暂时不应跳进去的难能力

- 会把项目从硬件产品变成复杂系统或生态。
- 需要医疗、算法、云服务、订阅内容或大规模长期运营能力。
- 例如医疗光疗宣称、复杂睡眠算法、强云依赖、订阅内容生态、传感器闭环睡眠监测。

## 4. 爆发力 USP 生成流程

```text
评论证据 / 竞品购买理由
-> 用户真实任务
-> 一眼有画面的场景事件
-> 爆发力 USP 候选
-> 能力成长边界检查
-> Red-team argue
-> 技术拓扑：用户任务效果 -> 功能条件 -> 关键元器件
-> Human Decision Stop
```

## 5. USP 不是利益点列表

不要把常规利益点直接当主 USP，例如：

- 更好看
- 更可靠
- 无订阅
- 更便宜
- 更好用
- 更多模式

这些通常是支撑项。主 USP 应该被压缩成一个消费者能立刻想象的场景事件。

示例：

```text
普通方向：像灯具一样的日出光效
更强命题：先叫醒房间，再叫醒你

普通方向：无订阅完整 routine
更强命题：按一下晚安，明早自然醒

普通方向：亮度更柔和
更强命题：不是闹钟响了，是卧室天亮了
```

## 6. 爆发力 USP 评分矩阵

默认 100 分制：

| 维度 | 分值 | 判断问题 |
|---|---:|---|
| 画面冲击 | 25 | 用户看到主图/视频第一秒，能不能立刻想象自己的生活场景发生变化？ |
| 证据支撑 | 20 | 评论、竞品销量、竞品品牌心智是否证明这个需求存在？ |
| 缺口强度 | 15 | 竞品是否证明需求成立，但交付方式仍有明显不满？ |
| 能力成长匹配 | 20 | 是否落在核心能力或可控相邻成长能力内？ |
| 前台表达 | 10 | 是否能用主图、视频、A+、五点描述讲清？ |
| 落地难度反向 | 10 | 是否不需要跳进高风险复杂系统？ |

注意：

- 评分不是事实，是 AI 判断辅助。
- `证据支撑` 和 `缺口强度` 必须来自真实证据。
- `画面冲击`、`能力成长匹配`、`前台表达`、`落地难度反向` 必须标注为推断。
- 最终选择必须停在人类判断。

## 6.1 评论频次与 USP 权重校准

第 6 章不能直接把评论主题频次当成主 USP 排名。评论频次只能说明样本中哪些话题被更多提及；USP 权重要回答公司应该押注哪条产品赛道。

必须拆成两张表或两个图：

```text
评论提及频次
-> 用户在评论里反复说了什么
-> 支撑：raw / tagged / effective review ledger

USP 战略权重
-> 哪个命题最值得进入产品规划
-> 支撑：购买驱动、竞品缺口、公司能力、第一眼表达、技术拓扑、BOM/test 可验证性
```

如果某个主题评论频次很高，但只是保障性价值、审美支持项或使用体验配套项，不能自动升为主 USP。它只能作为主 USP 的支撑模块，除非 red-team 证明它同时具备购买冲动、竞争缺口、前台表达和可落地架构。

所有 review-derived 数字必须可复算：

- 至少保留 `process/*review*raw*ledger*.json`、`process/*review*tagged*ledger*.json` 或 `process/*review*effective*ledger*.json`；
- ledger 中必须能看到 review row 或等价字段，例如 `review_id`、`content_text`、`rating`、`asin`；
- 如果只有摘要，必须写 `not_recomputable`，不能把数字写成验证事实。

## 7. Red-team 立场

输出主 USP 前，至少从以下立场反驳：

1. 这是不是只是好文案，不是产品能力？
2. 这个 USP 是否要求公司跳进过难的新系统？
3. 用户是否真的会为这个场景买单，还是只是我们想象？
4. 竞品如果 4-6 个月复制，公司的优势还剩什么？
5. 这个命题能否反推到具体元器件、测试计划、BOM 和工业设计？
6. 前台表达是否一眼可见，还是必须靠长解释？

## 8. 技术拓扑输出

爆发力 USP 页面必须包含一张三层拓扑：

```text
用户任务效果
-> 功能性必须具备的条件
-> 关键元器件与结构发力点
```

每层必须回答：

- 用户要感受到什么？
- 要实现这个感受，功能必须满足什么条件？
- 哪些关键元器件、结构、工艺、软件或测试会决定成败？

## 9. Human Decision Stop

默认选择框：

```text
A. 选择推荐爆发力 USP，进入产品架构
B. 先做前台表达验证
C. 先做样机/关键技术可行性验证
D. 暂停，继续挖更锋利命题
```

AI 不得把推荐 USP 写成人类最终选择。

## 10. Wake Up Light 示例

在 wake up light 案例中，v12 结论“像灯具一样的日出光效与床头对象感”偏稳妥。v13 按本框架重做后，推荐命题变成：

```text
先叫醒房间，再叫醒你。
一盏把卧室调成早晨的睡醒灯。
不是闹钟响了，是卧室天亮了。
```

该命题仍以灯具光学、结构、CMF 和品质制造为根，但允许公司拓展空间光效曲线、本机 routine、声音模块和实体交互等相邻能力。
